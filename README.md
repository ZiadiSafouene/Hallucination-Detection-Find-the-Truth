# Hallucination Detection and Find the Truth

Python solution for Arabic hallucination detection and answer selection.

This project is built for **HalluScoring 2026 — Track 2: From Hallucination Detection to Truth**. It supports local prediction, local evaluation, and Codabench-style scoring.

### Overview

The pipeline has two main goals:

- Detect whether a generated Arabic answer contains hallucinated, unsupported, misleading, or factually incorrect information.
- Select the correct answer option when the generated answer is hallucinated.

### Objective

Given a question and an LLM-generated answer, the model must decide whether the answer is hallucinated.

Then, it must select the correct answer from six very similar options written in the same style as the LLM-generated answer.

The goal is to build models that can detect hallucinations and choose the correct answer among close and confusing options.

### Task 2.1 — Two-step Evaluation

#### Step 1: Hallucination Detection

Given a question and an LLM-generated answer, the model must determine whether the answer contains a hallucination.

This is a binary classification task.

Possible labels:

- `hallucination`
- `no_hallucination`

#### Step 2: Correct Answer Selection

If the generated answer is hallucinated, the model must select the correct answer from six closely written candidate options.

The six options are intentionally very similar and confusing. They are written in the same style as the LLM-generated answer, so the model must carefully identify the most accurate answer.

### Input Format

Each input example contains:

- `id`: the unique identifier of the example.
- `question`: the Arabic question.
- `generated_answer`: the answer generated by an LLM.
- `options`: six candidate answers.

#### Input Example

```json
  {
   "id": "islamicHallucination_9",
    "question": "ما هي السورة القرآنية التي فيها موضعان  للسجود؟",
    "generated_answer": "سورة الحج، الآيتان 18 و29.",
    "gold_answer": "سورة الحج، وموضعا السجود فيها في الآيتين 18 و77",
    "generator_model": "QCRI/Fanar-1-9B-Instruct",
    "label": "hallucination",
    "hallucinations": [
      {
        "span_start": 23,
        "span_end": 25,
        "hallucinated_span": "29",
        "explanation": "الإجابة الأساسية صحيحة؛ السورة التي فيها موضعان للسجود هي سورة الحج، لكن موضع السجدة الثانية ليس الآية 29، بل الآية 77."
      }
    ],
    "options": {
      "A": "السورة القرآنية التي فيها موضعان للسجود هي سورة الحج، وموضعا السجود فيها في الآيتين 18 و77.",
      "B": "سورة الحج، الآيتان 18 و29.",
      "C": "السورة القرآنية التي فيها موضعان للسجود هي سورة السجدة، لأن اسمها يدل على السجود.",
      "D": "السورة القرآنية التي فيها موضعان للسجود هي سورة العلق، لأنها ختمت بالأمر بالسجود.",
      "E": "السورة القرآنية التي فيها موضعان للسجود هي سورة النجم، لأنها من سور سجود التلاوة.",
      "F": "السورة القرآنية التي فيها موضعان للسجود هي سورة فصلت، لأن فيها آية سجدة مشهورة."
    },
    "answer": "A"
  },
```

### Output Format

Each prediction must contain:

- `id`: the example identifier.
- `label`: the hallucination detection label.
- `correct_option`: the selected correct option.

#### Output Example for a Hallucinated Answer

```json
{
  "id": "example_001",
  "label": "hallucination",
  "correct_option": "C"
}
```

#### Output Example for a Non-hallucinated Answer

```json
{
  "id": "example_002",
  "label": "no_hallucination"
}
```

### Expected Labels

For hallucination detection:

```text
hallucination
no_hallucination
```

For correct answer selection:

```text
A
B
C
D
E
F
```

From the repository root:

## Configuration

The main configuration file is:

```text
configs/config.yaml
```

It controls:

- the experiment name and output directory
- which domains are enabled
- which data files are used
- which model is enabled by default
- evaluation and Codabench settings

Example:

```yaml
experiment:
  name: "fanar_v1"
  output_dir: "outputs"
  log_dir: "logs"

models:
  "QCRI/Fanar-1-9B-Instruct": true
  "humain-ai/ALLaM-7B-Instruct-preview": false

domains:
  islamic:
    enabled: true
    data_dir: "/path/to/data/islamic/dev_subtask3-1_gold.json"
  general_culture:
    enabled: false
    data_dir: "/path/to/data/general_culture/dev_subtask3-2_gold.json"
```

Before running predictions, make sure each enabled domain points to an existing JSON file on your machine.

The model registry is stored in:

```text
configs/model_registry.yaml
```

It currently defines:

- `fanar`: `QCRI/Fanar-1-9B-Instruct`
- `allam`: `humain-ai/ALLaM-7B-Instruct-preview`


## Run Prediction

To run prediction with the default enabled model and enabled domain from `configs/config.yaml`:

```bash
python3 -m src.main predict --config configs/config.yaml
```

This is the main command for generating predictions.

The command will:

- read `configs/config.yaml`
- select the enabled model from the config
- load the enabled domain data
- generate predictions example by example
- save intermediate results every 20 examples by default
- resume from an existing prediction file if one already exists

The output file is written under:

```text
outputs/<experiment_name>/predictions/
```

For the default experiment name, the Islamic prediction file is:

```text
outputs/fanar_v1/predictions/prediction_islamic_fanar.json
```

## Choose a Model Explicitly

You can override the model from the command line:

```bash
python3 -m src.main predict --config configs/config.yaml 
```



## Resume Behavior

Prediction resumes automatically by default. If an output file already exists, the pipeline skips examples that already have predictions.

To force a fresh prediction run without resuming:

```bash
python3 -m src.main predict --config configs/config.yaml --no-resume
```

To change how often partial results are saved:

```bash
python3 -m src.main predict --config configs/config.yaml --save-every 10
```

## Codabench Format

To generate predictions in the Codabench-style format:

```bash
python3 -m src.main predict --config configs/config.yaml --codabench-format
```

The Codabench-style rows contain:

```json
{
  "id": "example-001",
  "pred_label": "hallucination",
  "pred_option": "A"
}
```

## Local Evaluation

After generating predictions, evaluate them against a gold file:

```bash
python3 -m src.main evaluate \
  --config configs/config.yaml \
  --prediction outputs/fanar_v1/predictions/prediction_islamic_fanar.json \
  --domain islamic
```

You can also provide a gold file explicitly:

```bash
python3 -m src.main evaluate \
  --config configs/config.yaml \
  --prediction outputs/fanar_v1/predictions/prediction_islamic_fanar.json \
  --domain islamic \
  --gold data/islamic/dev_subtask3-1_gold.json
```

The evaluator reports:

- detection accuracy
- detection F1 for the `hallucination` class
- macro F1 as an additional diagnostic metric
- option accuracy on all valid predictions
- combined score

The default Codabench-style score is:

```text
combined_score = 0.5 * detection_f1 + 0.5 * option_accuracy
```

`macro_f1` is reported as an extra metric, but it is not used in the official combined score unless the competition scorer changes.

## Codabench Scoring

To score a Codabench-style prediction file:

```bash
python3 -m src.main codabench \
  --prediction-path /app/input/res/prediction.json \
  --gold-path /app/input/ref/gold.json \
  --output-scores-path /app/output/scores.json
```

## Common Issues

### No enabled domains found

Check `configs/config.yaml` and make sure at least one domain has:

```yaml
enabled: true
```

### Data file not found

Update the `data_dir` value for the enabled domain in `configs/config.yaml`.

### CUDA or memory errors

Use a smaller model, reduce batch size in `configs/model_registry.yaml`, or switch the model device if your environment does not have enough GPU memory.

### Hugging Face access errors

Some models may require accepting terms or logging in to Hugging Face:

```bash
huggingface-cli login
```

## Notes

- The current prompt asks the model to return a strict JSON object with `predicted_label` and `selected_option` in all cases.
- Span-level error localization is not enabled in this version.
- The repository is structured so the prompt and parser can be extended later without changing the main CLI.
